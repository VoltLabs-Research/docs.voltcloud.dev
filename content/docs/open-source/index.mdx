## Overview

The open-source side of VOLT makes more sense when you stop looking at it as a flat list of repositories.

It is really a layered ecosystem. Some repositories implement scientific algorithms, some provide the native infrastructure that the daemon relies on, some expose SDKs for automation, and some are the applications your team interacts with directly.

That is why the docs site sits next to all of them. It is documenting a stack, not a single repo.

## The ecosystem in layers

### Foundational tools

These repositories provide the shared runtime and native capabilities the rest of the ecosystem builds on.

| Repository | Role | Links |
|---|---|---|
| **CoreToolkit** | Shared C++ foundation for math, geometry, simulation-cell handling, serialization, and analysis helpers used by scientific plugins | [GitHub](https://github.com/VoltLabs-Research/CoreToolkit) |
| **LammpsIO** | High-performance native parser for LAMMPS-oriented data used by the daemon | [GitHub](https://github.com/VoltLabs-Research/LammpsIO) · [npm](https://www.npmjs.com/package/@voltstack/lammps-io) |
| **SpatialAssembler** | Native GLB assembly for turning structured outputs into 3D assets | [GitHub](https://github.com/VoltLabs-Research/SpatialAssembler) · [npm](https://www.npmjs.com/package/@voltstack/spatial-assembler) |
| **HeadlessRasterizer** | Native raster pipeline for turning GLB assets into PNG previews | [GitHub](https://github.com/VoltLabs-Research/HeadlessRasterizer) · [npm](https://www.npmjs.com/package/@voltstack/headless-rasterizer) |

## How the pieces connect in practice

The easiest way to understand the ecosystem is to follow one analysis.

You upload a trajectory in Volt. The daemon parses it with native tooling, stores the dumps and derived assets, then executes a scientific plugin binary built on top of shared libraries such as CoreToolkit. The results come back as structured data, and VOLT or the SDKs consume that data as listings, charts, or 3D artifacts.

So the ecosystem is not a shelf of unrelated repos. It is a pipeline from raw simulation data to workspace-visible results.

## A note on the scientific algorithms

The foundational algorithms are developed independently and are not promised to be byte-for-byte replicas of OVITO or any other external implementation. In many cases the results line up closely, but for more involved algorithms you should expect implementation-specific behavior, parallelization differences, and the occasional numerical divergence that comes with independent scientific software.

That does not make the outputs untrustworthy. It means they should be understood as part of a serious software stack with its own assumptions and engineering tradeoffs.

## Contributing

Contributions are welcome across the ecosystem, but the kind of contribution matters. Documentation, SDK ergonomics, product improvements, and tooling are usually approachable entry points. The foundational algorithms are more specialized and deserve extra care, review, and domain understanding.

<Callout type="warn">
Do not treat the scientific repositories like generic CRUD code. Many of them implement subtle numerical and geometry-heavy algorithms where small changes can have outsized consequences.
</Callout>
